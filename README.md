# multi-label классификация для 41 финансового продукта клиентов банка
 
# Описание проекта
Цель проекта — разработать модель, предсказывающую вероятность открытия каждого из 41 банковского продукта (счета, карты, услуги) для заданного клиента. 
В отличие от стандартных рекомендательных систем, которые выдают топ-N продуктов, бизнесу требуются именно количественные оценки вероятности.

Банк располагает данными о 1 млн клиентов, из которых 750 тыс. используются для обучения, а 250 тыс. — для тестирования. 
Каждый клиент может владеть любым подмножеством из 41 банковского продукта. 
Данные полностью анонимизированы и обфусцированы: нет реальных названий продуктов и признаков.

Признаки:

Базовый набор — 200 признаков (категориальные cat_feature_i, числовые num_feature_j).

Дополнительный набор — более 2000 признаков (только числовые).

В признаках присутствуют пропуски и выбросы.

Целевая переменная — мультилейбл (41 бинарная метка, указывающая, открыт ли продукт у клиента).
Метрика соревнования — Macro Averaged ROC-AUC.

# Общий вывод

Проведен первичный анализ базового и дополнительного набора признаков.

На основе оценённой важности признаков были отобраны две подгруппы: 1300 признаков, суммарно дающих 90% важности, и 1600 признаков — 95% важности.»
Обучение проводилось на обеих подгруппах.

В решении использовались модели:

- CatBoostClassifier,
- multi‑label модель на основе HistGradientBoostingClassifier,
- multi‑label модель на основе XGBoost XGBClassifier,
- SketchBoost от pyboost.

Отдельно подбирались гиперпараметры модели HistGradientBoosting с помощью Optuna.

Лучшие значения макро‑усреднённой ROC‑AUC на тестовой выборке (250 тыс. клиентов) показали:

SketchBoost на 1300 признаках — 0.845678;

HistGradientBoosting (multi‑label) на 1600 признаках — 0.840862.

data/raw/ — сюда кладутся исходные данные в том виде, в каком ты их получил. Например, house_price_predict.csv с 3000 строками объявлений. Эти файлы никогда не меняются руками: если нужно что-то исправить, делаешь это в коде предобработки, а raw-данные остаются нетронутыми. Это гарантирует, что ты всегда можешь вернуться к началу и воспроизвести весь пайплайн с нуля. data/processed/ — результаты предобработки. Сюда попадают CSV после фильтрации выбросов, заполнения пропусков, создания новых признаков. Если у тебя есть несколько версий обработки, можно называть файлы с датой или версией: house_features_v1.csv, house_features_2024_01_15.csv. Это помогает отследить, какой именно препроцессинг дал лучший результат. models/ — обученные модели. CatBoost и LightGBM сохраняются в разные форматы (.cbm и .txt соответственно), поэтому удобно разделить их по подпапкам. Название файла модели должно содержать хотя бы базовую информацию: catboost_baseline_2024_01_15.cbm, lightgbm_tuned_rmse_12500.txt. Через месяц ты забудешь, что означает model_final.cbm, а расшифровывающее имя сэкономит время. notebooks/ — Jupyter-ноутбуки для экспериментов и визуализаций. Первичный анализ данных, построение графиков, быстрые проверки гипотез — все это живёт здесь. Ноутбуки удобны для исследования, но плохи для Production-кода: их сложно версионировать (Git показывает diff в JSON-формате) и нельзя нормально покрыть тестами. src/ — переиспользуемый код. Когда ты понял, как работает предобработка в ноутбуке, переноси логику в src/preprocessing/clean_data.py. Когда обкатал обучение модели, пиши src/training/train_catboost.py. Так код становится модульным: можно запускать скрипты из командной строки, писать на них unit-тесты, импортировать функции в разных местах проекта. logs/ — текстовые логи обучения, метрики экспериментов, трекинг параметров. Если используешь MLflow или Weights & Biases, они создают свои служебные папки — их тоже можно положить сюда или в отдельную experiments/. tests/ — pytest-тесты для функций из src/. На начальном этапе можно обойтись без них, но когда проект растёт, тесты спасают от регрессий: изменил функцию заполнения пропусков — запустил тесты и сразу увидел, что сломалось. Теперь про разделение raw и processed. Принцип простой: raw-данные неприкасаемы. Если house_price_predict.csv содержит опечатку в названии признака или там есть строки с нулевой площадью, ты не правишь CSV руками.
